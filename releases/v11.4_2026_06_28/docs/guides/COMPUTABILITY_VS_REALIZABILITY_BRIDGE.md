# Computability vs Realizability Bridge

Status: companion implementation guide. Canon controls if conflict occurs.

## Purpose

This guide makes explicit a boundary already required by Reality Grounding and CSV: a system may compute, simulate, generate, or describe a possible state that cannot be responsibly selected, executed, or treated as ethical decision output.

## Core rule

**Computable is not selectable. Generated is not grounded. Executable is not ethical. Realizable is necessary but not sufficient.**

A model output, simulation, plan, score, tool result, or dashboard value is only a candidate artifact. It has no selection authority until it is routed through the relevant RippleLogic surfaces.

## Cascade mapping

| Distinction | RippleLogic handling |
|---|---|
| Computable vs grounded | RG/RSG decides whether the claim has reality-surface support. |
| Grounded vs rights-safe | RF/NCRC decides whether protected rights and interests remain non-compensatorily protected. |
| Rights-safe vs ruin-safe | TRC decides whether catastrophic, irreversible, lock-in, or ruin exposure is bounded. |
| Ruin-safe vs structurally viable | CSV decides whether the pathway can stand under dependency, containment, execution, monitoring, authority, and host-system constraints. |
| Structurally viable vs best residual option | RLS ranks residual welfare among selectable survivors only. |

## Implementation rule

Decision tools, agents, wrappers, workbooks, and public examples SHOULD record when a computed or generated option is rejected, narrowed, redesigned, or held as sensitivity-only because it is not grounded, not rights-safe, not ruin-bounded, not structurally viable, not authorized, or not yet rankable.

## No new gate

This guide does not add a sixth public cascade level. Realizability is handled by the combined discipline of RG and CSV, with RF/NCRC and TRC preserving the moral and tail-risk constraints that mere realizability cannot supply.

## Public explanation

For nontechnical readers: AI can imagine more than reality can carry. RippleLogic does not ask only what a system can generate. It asks what can be grounded, protected, ruin-bounded, structurally preserved, and ripple-scored.
