# Rights Threshold Governance Note v1.0

**Status:** companion validation and governance note. Canon controls if conflict occurs.

## Purpose

The Rights Floor is only as strong as the governance of its thresholds. This note defines how RF/NCRC thresholds should be proposed, justified, challenged, updated, and audited without weakening non-compensatory rights.

## Core principle

Rights thresholds are **governance commitments under evidence**, not hidden empirical facts and not preferences to be optimized away. A threshold expresses the minimum acceptable protected condition for a right-covered cell under the declared decision context.

## Threshold record

Any threshold used in a Tier 3 or public claim-bearing Tier 2 run SHOULD include:

| Field | Required content |
|---|---|
| right_id | LIFE, BODY, ECOL, LBTY, NEED, DIGN, PROC, INFO, or governed extension |
| covered_cells | Canonical rights-coverage cells affected |
| threshold_value | Numeric or categorical minimum used for NCRC |
| threshold_source | Canon default, legal source, domain standard, expert panel, affected-party process, or pilot calibration |
| floor_rationale | Why the threshold is protective enough for the right |
| under-protection risk | How a too-low threshold could fail protected stakeholders |
| over-blocking risk | How a too-high threshold could block all feasible options |
| review_status | unreviewed, internal reviewed, independent reviewed, pilot calibrated, externally validated |
| revision_trigger | evidence, incident, challenge, legal change, domain update, or calibration failure |

## Revision governance

Threshold revisions MUST NOT be used to make a failing option pass after the fact. A threshold change should be applied to a future run or to a rerun with the change declared before selection.

Threshold changes SHOULD require:

1. stated reason for revision;
2. affected-right analysis;
3. comparison against previous threshold;
4. sensitivity rerun;
5. record of who proposed and approved the change;
6. affected-party or challenger review when rights-sensitive;
7. versioned publication if used in a public or Tier 3 run.

## Anti-capture rule

A decision owner who benefits from lower thresholds cannot unilaterally lower the Rights Floor. Where threshold adjustment materially changes admissibility, an independent challenger or review body SHOULD evaluate the change.

## Calibration plan

Future RF validation should test:

- inter-rater agreement on whether threshold crossings occurred;
- sensitivity of selection to threshold variation;
- subgroup-specific false negative rates;
- whether thresholds detect rights-adjacent harms before they become obvious violations;
- whether Emergency Mode is triggered too often or too rarely.

## Plain-language rule

Do not move the floor to fit the option. Redesign the option to respect the floor.
